## 摘要

AI 手机智能体（AI Mobile Agent）是一款基于 Android 平台的智能助手应用。用户通过自然语言描述需求，应用调用大语言模型（LLM）将指令分解为结构化的多步操作计划，再通过 Android AccessibilityService 无障碍服务执行 UI 自动化操作，实现"说话即可操控手机"的体验。项目采用 Google Clean Architecture 五模块分层架构，使用 Kotlin + Jetpack Compose + Room + Hilt + OkHttp 技术栈，支持 DeepSeek/OpenAI/Anthropic/Google 等 7 种大模型。双模型 AI 审查修复了全部 19 个代码问题，代码量约 4500 行 Kotlin + 600 行 ArkTS，编译验证通过，核心功能在模拟器上测试通过。

**关键词**：Android、无障碍服务、大语言模型、智能体、自动化、Clean Architecture、代码审查

## 一、选题背景与意义

### 1.1 选题背景

智能手机已经成为日常生活和工作不可或缺的工具，用户每天需要在多个应用中完成大量重复性操作——发微信、查天气、设闹钟、导航、分享内容等。虽然语音助手（如 Siri、小爱同学）已经出现，但它们的能力通常局限于系统级操作，无法跨应用完成复杂任务链路。

2025-2026 年，大语言模型（LLM）技术的成熟为这一问题提供了新的解决方案。LLM 能够理解自然语言意图、进行逻辑推理和规划，配合 Android 无障碍服务（AccessibilityService）的 UI 操控能力，可以构建真正的"AI 手机智能体"——用户只需一句话，AI 自动完成多步操作。

### 1.2 选题意义

- **技术探索**：将 LLM Agent 范式应用于移动端，探索"观察-思考-行动-验证"循环在手机自动化中的可行性
- **工程实践**：完整经历 Clean Architecture 设计、多模块 Gradle 配置、Hilt 依赖注入、Room 数据持久化的开发流程
- **简历价值**：项目涉及 Kotlin 协程、Jetpack Compose、SSE 流式响应、AccessibilityService 等高级 Android 技术，具有较好的展示性

## 二、需求分析

### 2.1 功能需求

| 编号 | 功能 | 描述 |
|------|------|------|
| F1 | 自然语言聊天 | 用户可与 AI 自由对话，支持 SSE 流式输出和多轮上下文 |
| F2 | 任务规划 | AI 将用户指令分解为 JSON 结构化的操作步骤 |
| F3 | 自动执行 | 通过 AccessibilityService 执行 open_app/tap/type/swipe 等操作 |
| F4 | 安全保护 | 检测支付/密码页面自动停止执行 |
| F5 | 多模型切换 | 支持 DeepSeek/OpenAI/Anthropic/Google 等 7 种模型 |
| F6 | 独立 API Key | 每个模型独立配置 API Key |
| F7 | 任务历史 | 本地持久化保存所有任务，支持查看和重放 |
| F8 | App 能力目录 | 注册哪些 App 可被操控及其支持的操作类型 |
| F9 | 中止对话 | 流式输出过程中可随时中止 |
| F10 | 文件上传 | 支持上传文本文件让 AI 分析 |

### 2.2 非功能需求

- 流式响应延迟 < 100ms（chunk 间间隔）
- 步骤执行成功率 > 80%（已注册 App 的 open_app 操作）
- APK 大小 < 60MB
- 支持 Android 10.0+
- UI 采用暗色 Material 3 主题

## 三、技术方案

### 3.1 技术选型

| 类别 | 技术 | 选型理由 |
|------|------|----------|
| 语言 | Kotlin | Android 官方推荐，协程支持成熟 |
| UI 框架 | Jetpack Compose + Material 3 | 声明式 UI，开发效率高 |
| 依赖注入 | Dagger Hilt | 官方推荐，自动管理生命周期 |
| 数据库 | Room | Android 原生 ORM，Flow 响应式查询 |
| 网络请求 | OkHttp（直连） | SSE 流式需要底层字节流控制 |
| LLM 服务 | DeepSeek API | 性价比高，OpenAI 兼容格式 |
| JSON 解析 | org.json（内置） | 流式场景轻量，无需额外依赖 |
| 执行引擎 | AccessibilityService | 系统级 UI 操控，无需 root |
| 构建工具 | Gradle 8.11.1 + JDK 21 | 多模块构建 |

### 3.2 架构选择：Clean Architecture

采纳课程推荐的 Google 标准架构，结合 Clean Architecture 依赖反转原则，划分为 5 个模块：

```
app ──► domain ◄── data
 │                 │
 ├──► execution    │
 └──► ui ◄─────────┘
```

- **domain**：纯 Kotlin，定义 Model + Repository 接口 + UseCase
- **data**：Room + OkHttp + Retrofit，实现 domain 接口
- **execution**：AccessibilityService + StepExecutor 实现
- **ui**：Compose Screen + ViewModel (Hilt)
- **app**：入口 + Hilt DI 配置

## 四、系统设计

### 4.1 核心数据流

```
用户输入 → ChatViewModel → StreamingLLMClient (SSE)
    → LLM 返回: "chat" 模式 → 流式文本显示
           或  "task" 模式 → JSON 步骤解析 → Task → TaskPlanCard
    → 用户确认 → TaskProgressScreen → ExecuteTaskUseCase
    → OTAV 循环(Observe→Think→Act→Verify) → AccessibilityService
    → 成功/失败反馈
```

### 4.2 数据库设计

三张核心表：

**Task**：id、userCommand、llmRawResponse、intent、confidence、status、appsInvolved、createdAt

**Step**：id、taskId(FK)、orderIndex、actionType、targetApp、targetElement、params、status、errorMessage

**AppCapability**：id、packageName、appName、supportedActions(JSON)、resourceIdMap(JSON)

### 4.3 Agent 循环设计

参考 ApkClaw/Droidrun 的成熟模式，设计 OTAV 四阶段循环：

1. **Observe**（观察）：解析当前屏幕 UI 树，获取包名和可交互元素
2. **Think**（思考）：安全检查 + 前置条件验证 + 策略调整
3. **Act**（行动）：通过 StepExecutorFactory 分发到具体执行器
4. **Verify**（验证）：确认动作效果，失败则重试（最多 2 次），危险则立即停止

### 4.4 LLM 提示词设计

系统提示词包含两部分：
1. **角色定义**："你是智能手机助手，能聊天也能帮执行手机任务"
2. **能力目录**：动态注入已注册 App 列表，格式为"应用名(包名)"
3. **输出规范**：要求 task 模式输出 `{"mode":"task","intent":"...","steps":[...]}`

JSON 解析采用 4 级回退策略：Markdown 代码块 → mode 标记回溯 → 字面量搜索 → 全文兜底。

## 六、创新点

1. **LLM Agent 移动端落地**：将学术界 OTAV Agent 循环在 Android 上完整实现，打通自然语言到设备操控的全链路
2. **双模式架构**：同一 LLM 接口同时支持聊天和任务两种模式，通过 mode 字段动态路由
3. **4 级 JSON 容错**：针对 LLM 输出格式不稳定的问题，设计 Markdown→回溯→搜索→兜底的回退链
4. **SSE 流式节流**：50ms 节流 + callbackFlow send() 解决 Android 端 SSE 缓冲溢出问题
5. **跨平台同构**：Android (Kotlin) 和 HarmonyOS (ArkTS) 采用相同 Clean Architecture 分层，Domain 层逻辑几乎完全复用

## 九、总结与展望

### 9.1 完成情况

| 指标 | 目标 | 实际 |
|------|------|------|
| 代码行数 | ~3000 | ~4500 Kotlin + ~600 ArkTS |
| 平台 | Android | Android ✅ + HarmonyOS 基础版 ✅ |
| 执行器 | 4 | 6 (open_app/tap/type/swipe/search/share_to) |
| LLM 模型 | 1 | 7 (DeepSeek×2/OpenAI×2/Anthropic×2/Google×1) |
| 代码问题 | — | 19 个审查发现，全部修复 |
| Git 提交 | — | 30+ 次，全在 GitHub 可追溯 |

### 9.2 不足与改进方向

1. **多步骤复杂任务**：单步 open_app 已验证，跨 App 多步骤链路（如"发微信给xxx"）需更多执行器打磨
2. **UI 元素定位**：不同 App 资源 ID 差异大，需更智能的元素匹配策略
3. **屏幕视觉**：依赖 UI 树，游戏等场景需截图+多模态识别
4. **鸿蒙版本**：框架已搭建，待接入 AccessibilityAbility 和流式 SSE

### 9.3 展望

未来可扩展方向包括：基于用户习惯的个性化学习、多设备协同、端侧模型推理（隐私保护）、与系统语音助手深度集成。
